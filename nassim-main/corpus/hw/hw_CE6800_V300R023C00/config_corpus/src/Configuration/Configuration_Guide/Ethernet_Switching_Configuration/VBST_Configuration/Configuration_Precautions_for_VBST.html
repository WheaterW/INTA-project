1. In VBST networking, if the link type of an interface on the local device is set to access or QinQ, the link type of the interface on the remote device must also be set to access or QinQ.

2. If the link types of the interfaces at both ends are not access or QinQ, the PVIDs of the interfaces at both ends must be the same. Otherwise, the two devices cannot negotiate the VBST status and therefore loops cannot be prevented.

3. A hybrid interface can allow packets from multiple VLANs to pass through in untagged mode and the allowed VLAN ID may be different from the PVID. As a result, VBST negotiation may fail. To prevent this issue, enabling VBST on a hybrid interface is not recommended.

4. VBST-enabled devices and STP-enabled/RSTP-enabled devices may be deployed together on the live network. VBST BPDUs and STP/RSTP BPDUs have different formats. Therefore, VBST-enabled devices and STP-enabled/RSTP-enabled devices cannot communicate with each other. To implement interworking with STP-enabled/RSTP-enabled devices:

· For an access interface, a VBST-enabled device uses STP BPDUs (when it is connected to an STP-enabled device) or RSTP BPDUs (when it is connected to an RSTP-enabled device) to communicate with the remote device based on the VLAN to which the access interface belongs. In this way, topology calculation is performed according to STP/RSTP, which does not differentiate VLANs. Therefore, a spanning tree shared by VLANs is formed.

· For a trunk or hybrid interface:

(1) When a VBST-enabled device is connected to an RSTP-enabled device, the VBST-enabled device uses RSTP BPDUs to communicate with the remote device in VLAN 1 and uses VBST BPDUs with the Data field of RSTP BPDUs to communicate with the remote device in other VLANs.

(2) When a VBST-enabled device is connected to an STP-enabled device, the VBST-enabled device uses STP BPDUs to communicate with the remote device in VLAN 1 and uses VBST BPDUs with the Data field of STP BPDUs to communicate with the remote device in other VLANs.
