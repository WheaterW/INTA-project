DHCP Server

The DHCP server function is not configured together with the M-LAG function, but the two functions are not mutually exclusive. This is because the DHCP server function may be used on the device configured with M-LAG. For example, in the ZTP deployment scenario, M-LAG has been configured on the device before the deployment. During the deployment, the device only needs to assign IP addresses to clients.

When configuring the DHCP server function on the device configured with M-LAG, pay attention to the following points:

1. Ensure that sufficient addresses are reserved in the address pool of each M-LAG member. That is, the address pool of a single M-LAG device can meet the address requirements of all terminals, and the address ranges of M-LAG members' address pools cannot overlap. In this scenario, the address pool allocation efficiency is affected.

2. The lease renewal function of a client is unavailable. As a result, the client may obtain an IP address again.

3. The unicast function (the function of expecting the server to unicast response packets) of a client is unavailable.

4. During the M-LAG fault (restart), a client broadcasts a lease renewal packet, which may cause the address to change.

DHCP common

The CAR capability depends on the actual number of packets received by the local device per second.

When the rate of DHCP service packets exceeds the CAR limit, packet loss occurs. The normal packet rate is the same as the packet sending rate. However, the service networking has the following special scenario: The same packet reaches the same device through different paths (the receiving packet rate is a multiple of the packet sending rate). In the typical client-relay agent-server networking, packets from the client can reach the server through the relay agent at Layer 3 or Layer 2. In this case, the server receives two copies of the same packets. If the CAR capability of the server is verified based on the rate at which the client sends packets, the CAR limit is exceeded.
