The network types of the interfaces on both ends of an OSPFv3 neighbor relationship must be the same so that the interfaces can learn routes after the neighbor relationship is established. Otherwise, the neighbor relationship cannot be established and routes cannot be calculated.

You are advised to use the same network type at both ends.

1. Loop detection can be implemented for a maximum of 100,000 imported routes. After the number is exceeded, the LSAs do not advertise route advertisement extension information for loop detection.

2. The scenario where routes are imported to NSSAs is not supported.

3. Redistribute information cannot be advertised for summary routes.

4. After a routing loop is removed, the device cannot automatically exit the routing loop state. To exit the routing loop state and clear the alarm, you need to manually clear the alarm using a command.

5. If the redistribute IDs of different devices conflict, a loop alarm may be falsely reported.

6. Routing loop detection is not supported in scenarios where more than two re-advertisement nodes exist.

7. Loops may fail to be detected in scenarios where a looped route and a non-looped route advertised by multiple nodes form load balancing in a process.

8. If the original route source is withdrawn and a loop is triggered, self-healing is not supported.

9. Route import between public and private networks is not supported.

10. When detecting a routing loop, a re-advertisement node increases the cost of the route to be re-advertised to eliminate the loop. This may cause the traffic direction to be different from the expected path.

11. When the maximum cost is advertised after a routing loop is detected, the apply cost configuration in a route-policy does not take effect.

12. After this function is enabled, LSAs need to be added for each prefix to advertise extended TLVs, which greatly increases memory usage.
