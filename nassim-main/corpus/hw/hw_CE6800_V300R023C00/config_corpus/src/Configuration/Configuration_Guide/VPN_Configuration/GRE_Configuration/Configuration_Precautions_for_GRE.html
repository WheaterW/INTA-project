After the device receives GRE packets that are fragmented based on the GRE header, the device cannot identify other segments except the first one because the GRE header is followed by a data packet but not the inner IP header. As a result, the forwarding paths of these GRE packets may be different, causing packet mis-sequencing on the destination device.

When an IP packet enters a GRE tunnel and is encapsulated:

1. The DSCP value in the original IP header is mapped to an internal priority based on the DiffServ default profile. The internal priority is obtained based on the DiffServ domain profile bound to a Layer 3 inbound interface.

2. The DSCP value in the newly encapsulated IP header is mapped to an internal priority based on the DiffServ domain profile bound to the outbound interface. The internal priority is obtained based on the new DSCP value in the original IP header and the DiffServ default profile.

3. The qos phb marking dscp enable command must be run in the system view to ensure the correct priority mapping of packets. Otherwise, the queue that packets on the outbound interface enter is determined based on the DSCP value of the original packet defined in the default DiffServ domain profile. The DSCP value of the packets sent from the device may not be the priority specified in pipe mode, because priority mapping is performed twice.

When an IP packet leaves a GRE tunnel and is decapsulated:

The DSCP value in the inner IP header is mapped based on the internal priority and the DiffServ domain profile bound to the outbound interface. The internal priority is obtained based on the DSCP value in the outer IP header and the DiffServ default profile.
